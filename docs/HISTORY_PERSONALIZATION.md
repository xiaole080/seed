対象画面: `src/screens/HistoryScreen.tsx`（ルート `log`）
作成: product-manager / 2026-05-23
実装担当: frontend-engineer（本ドキュメントの受け入れ条件に沿うこと）

## 1. 目的とスコープ

- 目的: 利用者本人が「自分の生活リズム・気分・通所状況の傾向に気づける」こと。
- 医療診断ではない。「良い/悪い」評価ではなく「自分の傾向に気づく」ことが主眼。
- MVP段階。高度な分析・診断・AI予測は入れない。
- データはローカル保存のみを読む。Google Sheets / 外部API は一切読まない。

### 使用データ（読み取りのみ。書き込みは行わない）
- `seed.daily.v1` — `StoredDailyRecord`（`store.ts`）。`listDailyRecords()` で取得。
- `seed.attendance.v1` — `AttendanceMonthlyRecord`（`types.ts`）。`listAttendanceByMonth(yyyyMM)` で取得。
- `seed.app.state.v1` — `AppState.recordIds`（`App.tsx`）。HistoryScreen に props で渡す。

## 2. 現状把握（要約）

- `HistoryScreen.tsx` は現状、モックデータ `HISTORY_14`（`src/data/history.ts`）に固定依存している。
  実データ（`seed.daily.v1` / `seed.attendance.v1`）も `recordIds` も一切読んでいない。
- 上部サマリーカード3つ（記録した日 `11/14`、気分の平均 `3.5`、通所した日 `8`）はハードコードの固定値。
- タブは `mood / sleep / meds / attend` の4種固定。期間切替（7日/14日/今月）は存在しない。
- `MoodTrend` は折れ線を描くが、未記録日を空白にする概念がなく、全データ点を等間隔で結ぶだけ。
  `data.length - 1` で除算しているため、データ点が0〜1件のとき除算エラー/NaN になる。
- `MedsCalendar` のサマリー文「きちんと飲めています」、`AttendCalendar` の通所セルなど、固定文言。
- `HISTORY_14` の `HistoryEntry` 型は `StoredDailyRecord` とは別物。`primaryInfluence`・睡眠詳細・
  食事/運動/体調の各フラグ・自由記述 `note` などは現状のきろく画面では一切扱っていない。
- `App.tsx` の `route === 'log'` 分岐は `<HistoryScreen onTab={...} />` のみで、`recordIds` を渡していない。
  → props 追加が必要（タスク0）。

### 既存 `store.ts` で再利用できるもの
| 関数 | 用途 |
|---|---|
| `listDailyRecords()` | 全 `StoredDailyRecord` を date 昇順で取得。期間フィルタの入力にする |
| `listAttendanceByMonth(yyyyMM)` | 指定月の通所レコード取得。「今月」モードでそのまま使える |
| `getDailyRecord(date)` | 特定日の取得（カード詳細などで必要なら） |
| `getAttendance(date)` | 特定日の通所取得 |
| `todayISO()` / `thisMonth()` / `weekdayEnFor()` | 期間端の算出・曜日表示 |
| `scheduleSlotFor(date, schedule)` | 通所「予定」の算出（リスト/カレンダーの予定表示に使える） |

### 新規に必要なロジック（実装対象。`store.ts` には無い）
- 期間フィルタ: 「今日から過去N日」（7/14）と「今月（YYYY-MM）」で `StoredDailyRecord[]` / `AttendanceMonthlyRecord[]` を切り出す。
  - 7/14日はあくまで日付レンジ。`listAttendanceByMonth` は月単位なので、7/14日が月をまたぐ場合は前月＋当月の2回呼んでマージし、レンジで再フィルタする。
- 「記録した日」カウント: 期間内の `StoredDailyRecord` のユニーク日数。
- 「記録した日の平均」気分: 期間内で `mood` が記録された日のみを母数に平均（未記録日を0扱いしない）。
  `missingness.skippedMood === true` の日は母数から除外。
- 影響要因集計: 期間内 `StoredDailyRecord[]` の `primaryInfluence`（配列）を全件フラットに数え、頻度降順に並べる。
- 通所サマリー: 期間内 `AttendanceMonthlyRecord` の状態分類（後述）と件数集計。
- 記録項目別の傾向集計: 睡眠/食事/運動/体調/服薬の各カードに必要な軽い集計（カードごとに後述）。

新規ロジックは `HistoryScreen.tsx` 内のヘルパー、または `src/data/historyStats.ts`（新規・純関数のみ）に置くことを推奨。
**`store.ts` の既存関数のシグネチャは変更しない**こと（他画面が依存）。

## 3. タスク一覧

依存関係: T0 → T1 →（T2..T7 は T1 完了後に並行可）→ T8（横断確認）

### T0. HistoryScreen に実データ／recordIds を流す配線

- 概要: `App.tsx` の `route === 'log'` 分岐で `HistoryScreen` に `recordIds={state.recordIds}` を渡す。
  `HistoryScreen` の props 型に `recordIds: string[]` を追加。モック `HISTORY_14` 依存を外し、
  `listDailyRecords()` / 通所ストアからの読み取りに置き換える土台を作る。
- 依存: なし
- 受け入れ条件:
  - Given アプリが `route === 'log'` を表示する / When HistoryScreen がマウントされる /
    Then `seed.daily.v1` と `seed.attendance.v1` を読み、`seed.app.state.v1.recordIds` を props で受け取っている。
  - Given `seed.daily.v1` が空（記録ゼロ）/ When きろく画面を開く /
    Then 例外を投げずに描画され、各セクションは空状態（後述の責めない空文言）を表示する。
  - Given 改修後 / Then `src/data/history.ts`（`HISTORY_14` 等のモック）への参照が `HistoryScreen.tsx` から消えている。

### T1. 表示モード切替タブ「7日 / 14日 / 今月」

- 概要: 画面上部に期間切替タブを追加。選択期間が下の全可視化（サマリー・気分・影響要因・通所・項目カード）に反映される。
- 依存: T0
- 受け入れ条件:
  - Given きろく画面 / When 画面を開く / Then 上部に「7日 / 14日 / 今月」の3タブがあり、初期選択は「7日」。
  - Given いずれかのタブを選ぶ / When タップする / Then 選択状態が視覚的に変わり、配下の全可視化がその期間で再計算される。
  - Given「7日」「14日」/ Then 期間は「今日を含む過去7（14）日」。`todayISO()` を終端に算出する。
  - Given「今月」/ Then 期間は当月1日〜今日（`thisMonth()` 基準）。
  - Given 期間が月をまたぐ（例: 月初に「14日」）/ When 通所データを取得 / Then 前月＋当月をマージしレンジで絞り、欠落なく表示する。

### T2. 責めないサマリー一言

- 概要: 画面上部（期間タブ直下）に、利用者を責めない短いサマリー文を1行表示。
- 依存: T1
- 受け入れ条件:
  - Given 期間内に記録がある / When 画面を開く / Then 「今週は3日記録できています」等、記録できた事実を肯定的に述べる文を表示する。
  - Given 期間内の睡眠記録が少ない（または0）/ Then 「睡眠の記録が少ないため、まだ傾向は見えにくいです」のような、判断を保留する中立文を表示する。
  - Given どの期間でも / Then 禁止語（「記録できていません」「継続できていません」「さぼっています」「改善が必要です」「悪化しています」）および医療診断語（「うつ傾向」「双極傾向」「症状が悪化」「治療が必要」「再発リスク」「危険状態」）を一切含まない。
  - Given 記録ゼロ / Then 「まだ記録がありません。気が向いたときに記録してみてください」程度の、行動を強制しない文を表示する。
  - サマリー文は定数テーブルとして集約し、文言レビューしやすい形にする（privacy/PMが文言を一望できること）。

### T3. 気分の可視化（折れ線 or ドットチャート）

- 概要: 期間内の `mood`(1〜5) を時系列で可視化。記録がない日は空白（点を打たない）。
- 依存: T1
- 受け入れ条件:
  - Given 期間内に複数日の気分記録 / When 気分セクションを見る / Then 各記録日の `mood` が時系列に並ぶ折れ線またはドットで表示される。
  - Given 期間内に未記録日がある / Then その日は空白（点なし／線でつながない）。`mood` を 0 や 3 で補完しない。
  - Given 平均値を表示する / Then ラベルは「記録した日の平均」とし、母数は気分を記録した日のみ。未記録日を0として平均に算入しない。
  - Given 気分記録が0件 / Then 折れ線/ドットは描画せず「まだ気分の記録がありません」を表示し、平均は出さない（NaN を表示しない）。
  - Given 気分記録が1件のみ / Then 除算エラー（現状 `data.length - 1` 由来）を起こさず、点1つを安全に描画する。
  - Given 縦軸 / Then 「げんき / つらい」等の方向ラベルは残してよいが、良し悪し評価の断定文（例「気分が悪化」）は出さない。

### T4. 影響要因ランキング

- 概要: 期間内 `primaryInfluence` を頻度降順でランキング表示。
- 依存: T1
- 受け入れ条件:
  - Given 期間内に `primaryInfluence` を含む記録 / When 影響要因セクションを見る / Then 見出しは「最近、気分に関係していそうなこと」等の柔らかい表現。
  - Given 集計 / Then `primaryInfluence` 配列を全記録分フラット化して頻度を数え、降順に並べる。各項目は `PRIMARY_INFLUENCES`（`moods.ts`）の `label` と `icon` で表示する。
  - Given 表示件数 / Then 上位5件程度まで（チップ/リストで可）。頻度が同数の項目は安定した順で表示する。
  - Given `primaryInfluence` が一度も記録されていない / Then 「まだ気分に関係することの記録がありません」のような中立文を表示する。
  - Given どの表示でも / Then 「この要因が悪い」「この要因が原因で症状が…」のような因果断定・診断表現を出さない（観察事実の提示にとどめる）。

### T5. 通所リズムの可視化（カレンダー風 or リスト）

- 概要: 期間内の `AttendanceMonthlyRecord` をカレンダー風またはリストで表示。
- 依存: T1
- 受け入れ条件:
  - Given 期間内の通所データ / When 通所セクションを見る / Then 各日が「通所予定」「通所済み（在宅含む）」「休み」「未打刻」のいずれかとして表示される。
  - 状態の判定（断定しない）:
    - 予定: `plannedMode`（`scheduleSlotFor` または レコードの `plannedMode`）から「通所/在宅/休み」。
    - 実績: `actualMode` と `checkIn`/`checkOut` の有無から「通所済み（チェックイン/アウト）」「在宅」を表示。
    - `missingClock === true` または打刻が欠けている日は「未打刻」「記録なし」と表示する。
  - Given 予定があり打刻がない日 / Then 「欠席」「休んだ」と断定せず、「未打刻」または「記録なし」と表示する。
  - Given 通所データが0件 / Then 「まだ通所の記録がありません」を表示する。
  - Given チェックイン/アウト時刻 / Then 時刻は表示してよいが、滞在時間の長短に対する評価（「短い」「もっと」等）は出さない。

### T6. ONの記録項目だけカード表示

- 概要: 「じぶん」画面でONの項目（`recordIds`）に対応するカードだけを表示。睡眠/食事/運動/体調/服薬。
- 依存: T1
- 受け入れ条件:
  - Given `recordIds` に `sleep` が含まれる / When きろく画面を見る / Then 睡眠カードが表示される。含まれなければ睡眠カードは表示されない。食事(`meal`)・運動(`exercise`)・体調(`condition`)・服薬(`meds`)も同様。
  - Given `recordIds` が空 / Then 項目カードは1つも表示されず、その旨の中立文（「表示する記録項目が選ばれていません」程度）を出す。
  - Given ONだが期間内にその項目のデータが0件 / Then カードは表示し、中身は責めない空文言（「まだ◯◯の記録がありません」）にする。
  - Given カード内容 / Then `RECORD_PRESETS` / `CATEGORIES`（`records.ts` / `moods.ts`）の `label`・`icon` を流用する。
  - 各カードの集計（傾向）は T7 の「できれば」に該当。本タスク（必須）はカードの出し分けと空状態までを範囲とする。

### T7.（できれば）記録項目別の傾向カード中身 + 自由記述折りたたみ + 月末振り返り

- 概要: 各項目カードに軽い傾向集計を入れる。自由記述の安全な扱い、月末サマリーを追加。
- 依存: T6
- 受け入れ条件:
  - 睡眠カード: 期間内の入眠/起床時刻の傾向（記録がある日のみ）、`sleepIssues` の頻出項目、`nightAwakenings` の分布を、責めない表現で要約する。記録がない日は集計母数に入れない。
  - 食事カード: 期間内の `mealStatus` 分布や食事を記録できた日数を中立に提示する。
  - 運動カード: `activityFlags` の頻出項目をチップ表示する。
  - 体調カード: `conditionFlags` の頻出項目を提示する。「特になし」も中立に扱う。
  - 服薬カード: 服薬を記録した日数・`medicationStatus` の分布を中立に提示する。**薬名・薬の詳細は一切表示しない**（`StoredDailyRecord` に薬名フィールドはない。新たに扱わない）。「飲み忘れ」を責める表現にしない。
  - 自由記述（`note`）: 一覧にそのまま露出させない。既定は非表示。本人が明示操作（折りたたみを開く／「読む」ボタン）したときのみ該当日の `note` を表示する。先頭抜粋やプレビューも既定では出さない。
  - 月末振り返りサマリー: 「今月」モードのとき、月全体の記録日数・気分の記録した日の平均などを責めない一言でまとめる。
  - 上記いずれの文言も T2 の禁止語ルールに従う。

### T8. 横断確認（実装後チェック8項目）

- 概要: 改修全体に対する受け入れ確認。frontend-engineer 自己チェック後、qa-tester が再検証する。
- 依存: T1〜T7
- 受け入れ条件（実装後の確認8項目）:
  1. `npm run build` が型エラー・ビルドエラーなく通過する。
  2. `npm run lint` が新規 warning/error を増やさない。
  3. 未記録日を気分平均の計算で 0 点扱いしていない（母数は記録日のみ／ラベルは「記録した日の平均」）。
  4. 自由記述（`note`）が一覧・サマリー・カードにそのまま露出していない（本人の明示操作で初めて表示）。
  5. 画面のどこにも医療診断語（「うつ傾向」「双極傾向」「症状が悪化」「治療が必要」「再発リスク」「危険状態」）が出ない。
  6. 画面のどこにも責める文言（「記録できていません」「継続できていません」「さぼっています」「改善が必要です」「悪化しています」）が出ない。
  7. 記録ゼロ・気分1件のみ・通所0件など空/極小データで例外（NaN表示・除算エラー・クラッシュ）が起きない。
  8. `recordIds` のON/OFFに応じて項目カードが正しく出し分けされ、外部送信コード（fetch/axios/sendBeacon等）を新規に追加していない（読むのは localStorage のみ）。

## 4. MVP外と判断したもの（今回やらない / 対象外）

依頼の「今回やらない」リストに沿い、以下は本改修のスコープ外とする。提案が出ても実装しないこと。

- AI予測・将来予測（気分の予測、再発予測など）— 対象外。MVPは「振り返り」であり予測機能を持たない。
- 医療的判定・診断ラベル付け（傾向の病名化、リスク判定）— 対象外。アプリは医療診断ではない。
- 支援員向け管理画面・支援員視点のダッシュボード — 対象外。本人向け画面のみ。
- 複数利用者の比較・平均との比較・偏差表示 — 対象外。比較は「良い/悪い」評価を生むため目的に反する。
- 研究用エクスポート / 新たな CSV・Sheets 出力項目 — 対象外。きろく画面は表示のみで出力機能を足さない。
- 外部API連携・サーバー保存・解析SDK — 対象外。データは localStorage 読み取りのみ。
- 新しい記録項目の追加 — 対象外。既存5項目（睡眠/食事/運動/体調/服薬）+ 気分/影響要因/通所/自由記述の範囲内で可視化する。
- 服薬の薬名・用量・薬剤詳細の表示 — 対象外。`StoredDailyRecord` は `medicationStatus` のみ保持。新フィールドを足さない。

## 5. privacy-reviewer レビュー要否

**要レビュー（CLAUDE.md「省略禁止」ケースに該当）。** 実装完了後、qa-tester の後に privacy-reviewer を必ず通すこと。

該当理由:
- 記録項目（気分・睡眠・服薬・通所・自由記述）の **表示** に関わる変更である。
- 自由記述（`note`）の取り扱い（折りたたみ表示・非露出）が privacy 上クリティカル。露出範囲の判断にレビューが要る。
- 服薬データの表示粒度（薬名を出さない、責めない表現）の確認が要る。
- データ保存先に関わる読み取り元（localStorage 3キー）の妥当性確認。

privacy-reviewer に特に見てもらう点:
- 自由記述が既定で非表示か、プレビュー/抜粋も出していないか。
- 外部送信コードを新規追加していないか（読むのは localStorage のみ、書き込み・送信なし）。
- 服薬・気分・影響要因の表示が断定的・評価的になっていないか。
- 禁止語（責める／医療診断）が画面・定数テーブルに混入していないか。

備考: 本改修は localStorage の **読み取りのみ**。`seed.daily.v1` / `seed.attendance.v1` / `seed.app.state.v1`
への書き込みは行わない。Google Sheets / CSV 出力の項目・挙動は変更しない。
